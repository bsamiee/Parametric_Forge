# [G2_OSS_ECOSYSTEM]

The Nix theming-module and estate-monitor axis for F03 (theme owner as design system) and F07 (estate dashboards): what the external ecosystem owns, its exact option/command surface, and a durable adoption verdict per candidate. The standards/token-tooling axis (DTCG, WCAG, type scale, HTML primitives) lives in `capabilities.md`, which also owns the full local-owner surface; this file carries only the baseline the ecosystem candidates measure against.

## [01]-[LOCAL_OWNER_BASELINE]

The surface every candidate below extends, feeds, or replaces — capability detail is in `capabilities.md §01`.

- [OWNERSHIP_TOPOLOGY]: one palette owner (`theme.nix`) emits machine-readable artifacts (`palette.json`, `forge-dracula.tmTheme`, `base24.yaml`, `palette.html`, `coverage.json`), one coverage registry records target adoption (`targetsProved`, fails eval on a missing owner path), one font owner (`fonts.nix`) emits a parity-checked manifest, and runtime seams fix the artifact paths consumers hardcode. This is already the advanced-OSS pattern the candidates embody. Source: `modules/home/theme.nix:399-460`, `:826-840`, `modules/home/fonts.nix:137-194`, `docs/atlas/interconnection.md:64-72`.
- [PALETTE_MODEL]: colors are records rendering `hex, r, g, b, triple, csv, rgba` from one row (not Base16/24 slots), so Forge's palette is richer than the slot-indexed schemes the Nix ecosystem trades in; `base24.yaml` is the egress adapter down to the `base00`–`base17` slot model, not the canonical owner. Source: `modules/home/theme.nix:18-31`, `:531-533`.
- [ROLE_AND_SYNTAX_MODEL]: `roles.{surface,text,accent,state,diff,git,ui}` bind meaning to source-verified group names, with glyph-bearing git states where meaning never rides hue alone; a single syntax pivot carries `name`/TextMate `scope`/`color`/`style`/Tree-sitter `captures`. Source: `modules/home/theme.nix:76-150`, `:169-310`.

## [02]-[NIX_THEMING_MODULES]

Candidate engines for the palette/scheme/target layer; each row folds capability delta + exact option spellings + adoption verdict.

- [STYLIX]: End-to-end Nix theming across NixOS, Home Manager, nix-darwin, Nix-on-Droid — platform modules, wallpaper-derived palettes, fonts, icon themes, opacity, overlays, Base16/24 egress, target auto-enrollment. Palette owner: `stylix.base16Scheme` (path | YAML string | attrset; default derives from `stylix.image`, throws when image is null), `stylix.image`, `stylix.imageScalingMode` (`stretch|fill|fit|center|tile`), `stylix.polarity` (`either|light|dark`), `stylix.override` (applied before `config.lib.stylix.colors` is generated). Enrollment: `stylix.enable`/`stylix.autoEnable`/`stylix.overlays.enable`. Fonts: `stylix.fonts.{serif,sansSerif,monospace,emoji}.{name,package}`, `stylix.fonts.sizes.{desktop,applications,terminal,popups}`. Also `stylix.icons.{enable,package,dark,light}`, `stylix.opacity.{applications,desktop,popups,terminal}`. Verdict: nix-community-owned, actively maintained, no tagged releases — consume from a flake-input pin, never a release tag. A target-enrollment engine that accepts an external scheme via `stylix.base16Scheme` (attrset) — Forge's `base24.yaml` is exactly such an input — but wholesale adoption cedes the projection topology Forge already owns; reference for the palette-owner/target-enrollment split, not a drop-in. Source: https://nix-community.github.io/stylix/options/platforms/home_manager.html, https://raw.githubusercontent.com/nix-community/stylix/master/stylix/palette.nix.
- [CATPPUCCIN_NIX]: Port-oriented module graph — global `catppuccin.enable`/`catppuccin.autoEnable` govern enrollment; `catppuccin.flavor` (`latte|frappe|macchiato|mocha`, default `mocha`), `catppuccin.accent` (default `mauve`), `catppuccin.sources` (lazy attrset, recursive-update default so one override does not delete the rest), `catppuccin.cache.enable` (adds Catppuccin Cachix substituter + key), `catppuccin.enableReleaseCheck` (validates nixpkgs/NixOS/HM release relationship). Per-port modules inherit global enablement/flavor (`catppuccin.btop.{enable,flavor}`, `catppuccin.zellij.{enable,flavor}`). Verdict: release-gated, latest `v26.05` (2026-05-30) with a `release-26.05` stable branch — a fixed-palette (Catppuccin, not Forge Dracula) port graph, useful only as the global-enrollment/per-port-default pattern, never as a palette source. Source: https://nix.catppuccin.com/getting-started/, https://raw.githubusercontent.com/catppuccin/nix/main/modules/global.nix.
- [BASE16_NIX]: Scheme-as-function-and-attrset engine — `config.scheme` themes as a function over templates and as an attrset over color slots; `config.lib.base16.mkSchemeAttrs`, `config.scheme.withHashtag`, `config.scheme.override`, `templateRepo`, `target`, `base16.nixosModule`. Loads Base16 and Base24, applies templates, selects template variations as data (`templateRepo`+`target`, e.g. the `recolor` Zathura variation), supports custom YAML/Nix schemes and multi-scheme use. Verdict: the most adoptable engine here — one scheme function drives tmTheme and base24 egress in place of Forge's separate hand-rolled projections; a real candidate for the projection layer. Source: https://github.com/SenchoPens/base16.nix.
- [NIX_COLORS]: Home Manager `colorScheme` owner — 220+ Base16 schemes, pure-Nix YAML conversion via `nix-colors.lib.schemeFromYAML`, `config.colorScheme.palette.base00`–`base0F`, `config.colorScheme.variant` (consumer-readable light/dark), `nix-colors.homeManagerModules.default`, `nix-colors.colorSchemes.dracula`, plus contrib functions (image-derived schemes, wallpaper, vim scheme, GTK theme). Verdict: ARCHIVED read-only 2026-04-24, last released Apr 2023 before archival — dead. A pattern source (`variant` as a consumer-readable value, `schemeFromYAML`), never an adoptable dependency. Source: https://github.com/Misterio77/nix-colors.
- [TINTED_SCHEMES]: Centralized scheme corpus with `base16` and `base24` families consolidated into one repository (spec 0.11 branch) after the split was retired to simplify builders; the scheme/style/template/builder model projects one palette into 70+ editor, terminal, shell, and application themes, with Base24 covering ANSI expansion beyond Base16. Verdict: the canonical scheme-exchange corpus Forge's `base24.yaml` egress targets — an interop reference and a ready source of schemes for the F03 one-row-variant unlock. Source: https://github.com/tinted-theming/schemes, https://github.com/tinted-theming/home, https://github.com/tinted-theming/base24.

## [03]-[ESTATE_DASHBOARD_TOOLING]

Precedents and data sources for the F07 estate dashboard; capability + exact surface + verdict.

- [GLANCES]: Cross-platform monitor (CPU, memory, disk, network, processes, users, sensors, containers) exposing one model through many egresses: terminal UI, client/server (`glances -s`, `glances -c @server`), web UI (`glances -w`, served on `http://<ip>:61208`, URL refresh like `/10`, responsive), RESTful + XML-RPC APIs, stdout (`--stdout`, `--stdout-csv`, `--stdout-json`), `--fetch` mode, Central Browser discovery (`--browser`), and MCP server (`--enable-mcp` mounts an MCP SSE server at `/mcp` alongside the web server, Glances 4.5.x). Central Browser column grammar is `<plugin>:<field>:(<key>)`, defaults `system:hr_name`, `load:min5`, `cpu:total`, `mem:percent`; auth via `--username`/`--password`. Verdict: the JSON/MCP data source an F07 dashboard folds as one more observation-only feed beside `forge-update-board` receipts — `--stdout-json` for the page, `--enable-mcp` for agents reading the same facts (the F07 dual-consumer shape); the column grammar is a tile-vocabulary precedent. Source: https://glances.readthedocs.io/en/latest/quickstart.html, https://github.com/nicolargo/glances.
- [BTOP]: Dense resource monitor — always-visible processor/memory/disk/network/process boxes, process selection with detailed selected-process stats, filtering, sorting, tree view, process signaling, paused lists, custom presets, and theme files with explicit search paths (`--themes-dir`). Renderer capability matrix distinguishes truecolor, 256-color fallback (`-lc`/`--low-color`), and 16-color TTY (`-t`/`--tty`), tying graph correctness to terminal glyph coverage for Braille, geometric shapes, box-drawing, and block elements; GPU support gated by `GPU_SUPPORT=false` / `cmake -DBTOP_GPU=false`. Verdict: an information-design precedent (always-visible boxes + selection + detail + filter/sort/tree/preset), not a data source; its theme-file format is the projection target for making the monitor match the Forge palette. Source: https://github.com/aristocratos/btop.
- [WTF]: Terminal personal-information dashboard where every tile is a module configured in `config.yml`; documented module families span cloud, source control, calendar, news, breach/security, incident, and task systems (GitHub, Google Calendar, HackerNews, NewRelic, OpsGenie, Security, Transmission, Trello). Verdict: rename-in-progress toward Tessera (last release 2026-06-30) — the tile-as-module `config.yml` grammar is the precedent for the html-studio dashboard tile vocabulary; not a data source. Source: https://github.com/wtfutil/wtf.

## [04]-[PATTERNS_WORTH_COMPOSING]

- [PALETTE_OWNER_TARGET_ENROLLMENT_SPLIT]: Stylix separates palette admission (`stylix.base16Scheme`/`image`/`polarity`/`override` → `config.lib.stylix.colors`) from target projection (`mkEnableTarget`/`mkEnableTargetWith`), and protects the generated palette from GC when it is the default. Forge's `targetsProved` registry is already the eval-time target-enrollment analog; the untested seam is feeding Forge's `base24.yaml` into an external enrollment engine as `stylix.base16Scheme` while keeping palette ownership. Source: https://raw.githubusercontent.com/nix-community/stylix/master/stylix/palette.nix, https://raw.githubusercontent.com/nix-community/stylix/master/stylix/target.nix, `modules/home/theme.nix:453-460`.
- [GLOBAL_ENROLLMENT_PER_PORT_DEFAULT]: catppuccin/nix governs enrollment globally (`catppuccin.enable`/`autoEnable`) while per-port modules inherit global flavor, and keeps upstream port data in `catppuccin.sources` under recursive-update semantics so one override never deletes the remaining defaults — the inherit-with-per-port-override pattern for a multi-consumer theme graph. Source: https://raw.githubusercontent.com/catppuccin/nix/main/modules/global.nix.
- [SCHEME_AS_FUNCTION_AND_ATTRSET]: base16.nix treats one scheme as both a function over templates and an attrset over color slots (`config.scheme` themes Zathura/Neovim as a function, Alacritty via `config.scheme.withHashtag`), and records template variation as data through `templateRepo`/`target`. This is the direct route to driving Forge's tmTheme and base24 egress from one scheme value instead of separate hand-rolled projections. Source: https://github.com/SenchoPens/base16.nix.
- [ONE_MODEL_MANY_EGRESS]: Glances exposes a single monitoring model through terminal, client/server, web, RESTful, XML-RPC, stdout text/CSV/JSON, fetch, browser discovery, and MCP — the exact multi-egress-from-one-source shape F07 wants, where the page and the agents consume the same JSON. Source: https://glances.readthedocs.io/en/latest/quickstart.html.
- [TILE_AS_MODULE]: WTF makes every dashboard tile a `config.yml` module, and btop makes every box a configurable always-visible region with selection/detail/filter/tree — the tile-composition grammar the html-studio dashboard tile vocabulary mirrors. Source: https://github.com/wtfutil/wtf, https://github.com/aristocratos/btop.
- [FORGE_ALREADY_MATCHES]: Forge holds the advanced-OSS ownership pattern at the theme seam — one owner emits machine-readable palette artifacts, one coverage registry records target adoption, one font owner emits a manifest, runtime seams fix the artifact paths — so the ecosystem candidates are pattern references and interop egresses, not architectures to import. Source: `modules/home/theme.nix:399-460`, `:826-840`, `modules/home/fonts.nix:137-194`, `docs/atlas/interconnection.md:64-72`.

## [05]-[GAPS]

The next research wave must hunt:

- Whether Stylix or catppuccin can act as pure target-enrollment engines while an external owner keeps palette ownership — the `stylix.base16Scheme` path/attrset input is the exact seam to prove Forge can drive enrollment without ceding its projection topology.
- base16.nix `mkSchemeAttrs`/`templateRepo`/`target` exact API and closure cost for driving Forge's tmTheme + base24 egress from one scheme function — the concrete collapse of hand-rolled projections.
- Glances config-file grammar for Central Browser columns, the RESTful field-path map, and the `/mcp` SSE payload shape — needed to fold glances into the F07 observation-only substrate as a real feed.
- The DTCG bridge gap: no Nix theming module here projects DTCG token JSON — the Nix ecosystem is Base16/24-only — so whether Forge is the bridge from its Base24 egress to the DTCG token world is unmapped and owner-unclaimed.
- btop and glances theme-file formats: whether either consumes Base24 directly or needs a bespoke template — the projection target for palette-matched monitors on the estate.
- The post-rename Tessera module-config grammar (WTF's successor) as the settled tile-vocabulary precedent for the dashboard, since WTF itself is mid-rebrand.
