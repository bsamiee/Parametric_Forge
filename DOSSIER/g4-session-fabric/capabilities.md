Estate current-state truth for the G4 session fabric plus the native capabilities of the tools it already runs — Zellij, WezTerm, Yazi VFS — and the macOS mount substrate. External OSS candidates, session-manager grammars, and stealable patterns live in `ecosystem.md`.

## [02]-[ZELLIJ_NATIVE_CAPABILITY]

- Session resurrection is built in and on by default: each session serializes to a human-readable layout file in the system cache every second, recreating after intentional quit or crash. `pane_viewport_serialization = true` extends serialization to the viewport, and `scrollback_lines_to_serialize = 0` then serializes all scrollback. Root controls: `session_serialization`, `pane_viewport_serialization`, `scrollback_lines_to_serialize`, `post_command_discovery_hook`, `serialization_interval`, `disable_session_metadata`, `session_name`, `attach_to_session`. Source: https://zellij.dev/documentation/session-resurrection.html, https://zellij.dev/documentation/options.html.
- Resurrectable sessions list via `zellij list-sessions`/`ls`, resurrect via `zellij attach <session>` (or `--force-run-commands`), and delete via `zellij delete-session`/`delete-all-sessions`; `zellij --layout session-layout.kdl` reloads a saved layout. Source: https://zellij.dev/documentation/session-resurrection.html.
- Background control: `zellij attach --create-background <session>`, layout via `options --default-layout <layout-or-path>`, and global `--session <session>` before any subcommand (`zellij --session <s> action ...`, `zellij --session <s> subscribe ...`); `zellij watch <session>` attaches read-only. Source: https://zellij.dev/documentation/cli-recipes.html, https://zellij.dev/documentation/controlling-zellij-through-cli.html.
- State inspection spellings: `zellij action list-panes --json|--all`, `list-tabs --json|--state --layout`, `current-tab-info --json`, `list-clients`, `query-tab-names`. Capture spellings: `subscribe --pane-id terminal_1 --format json`, `action dump-screen --pane-id terminal_3 --full [--ansi]`, `action edit-scrollback --pane-id terminal_3`, `action dump-layout`. Source: https://zellij.dev/documentation/cli-recipes.html, https://zellij.dev/documentation/cli-actions.html.
- Layout runtime: KDL applied via `zellij --layout <file|name|url>` at startup, from the layout directory, or as new tabs inside a session; `zellij action new-tab --layout <file>` and `action override-layout <file>` accept `--retain-existing-terminal-panes`, `--retain-existing-plugin-panes`, `--apply-only-to-active-tab`, and `--layout-string`. Source: https://zellij.dev/documentation/cli-recipes.html, https://zellij.dev/documentation/layouts.html.
- Plugin API exposes typed session/pane state instead of terminal scraping: `get_session_list`, `get_pane_info`, `get_tab_info`, `get_pane_cwd`, `get_pane_running_command`, `get_pane_scrollback`, `save_session`, `switch_session[_with_layout|_with_cwd|_with_focus]`, `rename_session`, `delete_dead_session`, `delete_all_dead_sessions`, `kill_sessions`. Source: https://zellij.dev/documentation/plugin-api-commands.html.
- Top-level command surface: `attach [session-name]`, `list-sessions`, `kill-sessions`, `kill-all-sessions`, `options`, `setup` (`--check`, `--clean`, `--dump-config`, `--dump-layout [LAYOUT]`, `--generate-completion [SHELL]`). Source: https://zellij.dev/documentation/commands.html.

## [03]-[WEZTERM_NATIVE_CAPABILITY]

- `SshDomain` fields: `name`, `remote_address`, `username`, `ssh_option`, `multiplexing` (`"WezTerm"` | `"None"`), `no_agent_auth`, `connect_automatically`, `timeout`, `default_prog`, `assume_shell`, `local_echo_threshold_ms`, `remote_wezterm_path`; domain names are unique across all domain types. `wezterm.default_ssh_domains()` derives rows from `~/.ssh/config` — plain SSH domains need no remote WezTerm, multiplexing domains do. Source: https://wezterm.org/config/lua/SshDomain.html, https://wezterm.org/config/lua/wezterm/default_ssh_domains.html.
- `unix_domains` defines mux server and client domains; documented domain config keys: `default_domain`, `default_mux_server_domain`, `default_workspace`, `ssh_domains`, `unix_domains`, `wsl_domains`, `default_ssh_auth_sock`. Source: https://wezterm.org/config/lua/config/unix_domains.html, https://wezterm.org/config/lua/wezterm.mux/spawn_window.html.
- `SwitchToWorkspace` accepts `name` + `spawn` (a `SpawnCommand` run only when the workspace is absent); `SpawnCommandInNewWindow` takes a `SpawnCommand`; `wezterm.mux.spawn_window{}` fields are `args`, `cwd`, `set_environment_variables`, `domain` (`{ DomainName = '…' }` selects a configured mux domain), `width`, `height`, `workspace`, `position`. Mux APIs: `get_workspace_names`, `get_window`, `get_domain`, `set_default_domain`, `spawn_window`. Source: https://wezterm.org/config/lua/keyassignment/SwitchToWorkspace.html, https://wezterm.org/config/lua/keyassignment/SpawnCommandInNewWindow.html, https://wezterm.org/config/lua/wezterm.mux/spawn_window.html.
- CLI: `wezterm cli list --format json` returns rows with `window_id`, `tab_id`, `pane_id`, `workspace`, `size`, `title`, `cwd`; `wezterm cli spawn` (`--cwd`, `--domain-name`, `--new-window`, `--workspace` [requires `--new-window`]) emits the new pane id; `wezterm connect <DOMAIN_NAME> [PROG]` (`--new-tab`, `--workspace`) attaches a configured mux domain. Source: https://wezterm.org/cli/cli/list.html, https://wezterm.org/cli/cli/spawn.html, https://wezterm.org/cli/connect.html.
- [ADDED] `default_ssh_auth_sock` is nightly-only, so the estate's `SSH_AUTH_SOCK`→1Password override cannot ride WezTerm's last stable (`20240203-110809-5046fc22`); on stable the mux server inherits the ambient `SSH_AUTH_SOCK`. Source (adjacency): https://wezterm.org/config/lua/config/default_ssh_auth_sock.html, https://github.com/wez/wezterm/releases/latest.

## [04]-[YAZI_VFS_AND_MOUNT_SUBSTRATE]

- Yazi `vfs.toml` grammar: `[services.<name>]` with `type`, `host`, `user`, `port`, optional `key_file`, `key_passphrase`, `password`, `cert_file`, `no_cert_verify`, `identity_agent`; the service enum is `type`-tagged and admits only the `Sftp(ServiceSftp)` variant. Addressed as `sftp://<service>` (`yazi sftp://<name>`, `cd sftp://<name>`, `reveal sftp://<name>//root/dog.jpg`). Source: https://yazi-rs.github.io/docs/configuration/vfs/, https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/config/sftp.rs, https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/config/service.rs.
- `ServiceSftp` has NO `cache` field; `identity_agent` defaults from `SSH_AUTH_SOCK` when that value is absolute, and an explicit value must be empty or an absolute path after sanitization. Auth order is password, key + cert, key, identity agent, then none — a successful connection opens an SSH session channel and requests subsystem `sftp`. Source: https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/config/sftp.rs, https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/provider/sftp/conn.rs.
- Yazi VFS is an in-process client, not an OS mountpoint: connections ride a `deadpool::managed::Pool` keyed by `ServiceSftp` with `max_size(8)`, a 45 s create timeout, a 60 s SSH inactivity timeout, and a 10 s keepalive. Only Yazi resolves `sftp://`; non-Yazi consumers get no POSIX path from `vfs.toml`. Source: https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/provider/sftp/conn.rs.
- [ADDED] F12's proposed `cache posture` mount-policy field has no Yazi sink: `ServiceSftp` exposes no cache knob, so a cache-posture row cannot project into `vfs.toml` and must either be dropped from the host-row schema or owned by a separate OS-mount lane (rclone/macFUSE, surveyed in `ecosystem.md`). Source (adjacency): https://raw.githubusercontent.com/sxyazi/yazi/main/yazi-vfs/src/config/sftp.rs, file:docs/FRONTIER.md:71.
- OS-level SSHFS is the POSIX-mount alternative Yazi's in-process VFS does not provide: `sshfs [user@]host:[dir] mountpoint` (`-oport=PORT`, BSD/macOS unmount `umount mountpoint`). But Homebrew `sshfs` is stable `3.7.6`, depends on `libfuse 3.18.2`, and is marked `Requires: Linux` — not a macOS install path. Source: https://github.com/libfuse/sshfs, https://formulae.brew.sh/formula/sshfs.
- macFUSE is the macOS FUSE substrate (not an SFTP client): `brew install --cask macfuse`, version `5.2.0`, requiring a kernel extension approved in Privacy & Security; `5.2.0` adds an FSKit backend and `MFMount.framework` for libfuse/libfuse3, updating libfuse3 to `3.18.2`. Source: https://formulae.brew.sh/cask/macfuse, https://github.com/macfuse/macfuse/releases/latest.
