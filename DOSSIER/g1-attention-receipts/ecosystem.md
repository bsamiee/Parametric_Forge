# G1 Attention Receipts Ecosystem

External candidates for F01/F02/F10/F11: capability deltas, exact option spellings, maintenance signals, and patterns worth composing. Estate-side truth lives in `capabilities.md`.

## [01]-[CANDIDATE_ROSTER]

- `zellij` 0.44.3 (release `v0.44.3`, 2026-05-13; matches the installed binary): delta is typed pane streaming through `subscribe`, JSON pane/tab inventory through `-j, --json`, pane-id-addressed focus, one-shot `dump-screen`, and plugin pipes with launch-on-first-message delivery. Sources: https://zellij.dev/documentation/zellij-subscribe.html ; https://zellij.dev/documentation/cli-actions.html ; https://api.github.com/repos/zellij-org/zellij/releases/latest
- `zjstatus` v0.23.0 (release 2026-04-18; matches the estate pin): delta is in-bar rendering through the `{notifications}`, `{pipe}`, and `{command_NAME}` widget families, regioned formats, `#[..]` style directives, and `color_` aliases; release assets are `zjstatus.wasm` and `zjframes.wasm`. Sources: https://github.com/dj95/zjstatus/wiki/4-%E2%80%90-Widgets ; https://api.github.com/repos/dj95/zjstatus/releases/latest
- `terminal-notifier` 2.0.0 (release 2017-11-01, upstream dormant since): delta is notification grouping, replacement, listing, removal, app activation, URL open, and click-command execution; hard limits are no sticky notifications and no action buttons in the 2.x line (the README routes both to `alerter`), `-sender` incompatible with `-execute`/`-activate`, and persistence controlled by the macOS per-app notification style, never per notification. Source: https://github.com/julienXX/terminal-notifier
- `alerter` 26.5 (release 2026-02-19; Swift/SPM rewrite, macOS 13.0+, double-dash CLI): delta over `terminal-notifier` is blocking alerts that stay until dismissed, `--reply` text capture, `--actions` buttons with dropdown, `--timeout`, scheduled `--at`, `--json` structured results, and `--ignoreDnd`; the process exits when the user interacts, so a notification becomes a synchronous typed answer channel. Constraint: `--actions` cannot combine with `--reply`. Source: https://github.com/vjeantet/alerter
- `ntfy` v2.26.0 (release 2026-07-09): delta is cross-device publish over HTTP POST or `ntfy publish`, with `Click` URLs, `Actions`, `Priority`, `Tags`, Markdown bodies, sequence-id notification mutation, and authentication — remote subscribers and mutable notification streams `terminal-notifier` cannot express. Sources: https://docs.ntfy.sh/publish/ ; https://api.github.com/repos/binwiederhier/ntfy/releases/latest
- `noti` 3.8.0 (Codeberg release 2025-03-07; the GitHub repository was archived 2026-05-01 and is not mirrored): delta is process-boundary wrappers — direct command wrapping, `--pwatch PID`, stdin message body via `-`, multi-backend routing including ntfy. Verdict: the Forge collector is already richer for session attention because it joins hook events with live tty/process state; noti stays a non-adoption. Sources: https://codeberg.org/roble/noti/raw/branch/main/README.md ; https://github.com/variadico/noti
- `jq` 1.8.x (release `jq-1.8.2`, 2026-06-20): delta is tolerant JSONL admission (`fromjson?`, estate-proven), raw-input, slurp, compact, unbuffered, and `--stream`/`--seq` incremental modes. Sources: https://jqlang.org/manual/ ; https://api.github.com/repos/jqlang/jq/releases/latest
- `DuckDB` v1.5.4 (release 2026-06-17; docs selector `1.5 (current)`; the Forge overlay pins the same version): delta is direct SQL over the JSONL receipt corpus — NDJSON table functions, schema inference, `union_by_name` over schema drift, virtual `filename` provenance (since 1.3.0), and JSON-to-nested transformation. Sources: https://duckdb.org/docs/current/data/json/loading_json ; https://api.github.com/repos/duckdb/duckdb/releases/latest ; `overlays/manifest.nix:18`
- `UNUserNotificationCenter`: the API ceiling above both CLI wrappers — Apple's UserNotifications framework owns notification requests, categories, actions, and text-input actions through `UNUserNotificationCenter`, `UNNotificationRequest`, `UNNotificationAction`, and `UNTextInputNotificationAction`; every capability the CLI candidates lack routes here. Sources: https://developer.apple.com/documentation/usernotifications/unusernotificationcenter ; https://developer.apple.com/documentation/usernotifications/unnotificationaction
- `viddy`: estate-proven watch renderer — `viddy --interval` receipt panels and `viddy --differences` git panels already run as `forge-zellij` floating rows. Source: `modules/home/programs/apps/zellij/ops.nix:22-40`

## [02]-[EXACT_OPTION_SPELLINGS]

- Zellij pane identity: `terminal_1`, `plugin_2`, or bare `N` equivalent to `terminal_N`, shared across `subscribe`, `dump-screen`, `edit-scrollback`, and `focus-pane-id`. Source: local `--help` on zellij 0.44.3.
- Zellij streaming: `zellij [--session <NAME>] subscribe -p terminal_3 -s 200 -f json --ansi`; `-s` bare delivers all scrollback; JSON output is NDJSON of `pane_update {pane_id, viewport, scrollback, is_initial}` and `pane_closed {pane_id}`. Sources: local `zellij subscribe --help`; https://zellij.dev/documentation/zellij-subscribe.html
- Zellij capture: `zellij action dump-screen --pane-id terminal_3 --full --ansi [--path <PATH>]` (stdout when `--path` omitted, focused pane when `--pane-id` omitted); `zellij action edit-scrollback --pane-id terminal_3 --ansi`. Source: local `--help` on 0.44.3.
- Zellij inventory: `zellij action list-panes --all --json` and `zellij action list-tabs --all --json`. Source: local `--help` on 0.44.3.
- Zellij pipe options: `zellij action pipe [-n NAME] [-a ARGS] [-p PLUGIN] [-c CONFIG] [-l] [-s] [-f BOOL] [-i BOOL] [-w CWD] [-t TITLE] [--] <PAYLOAD>`; a blank payload listens on STDIN. Source: local `zellij action pipe --help` on 0.44.3.
- Zellij pipe semantics: unnamed pipes receive a random UUID; pipes with no destination broadcast to all listening plugins; a destination is plugin URL plus configuration and loads the plugin on the first message; CLI-originated pipes support backpressure and plugin replies on the CLI pipe stdout; plugins receive a `PipeMessage` with source, source metadata, name, payload, arguments, and privacy status through the `pipe` lifecycle method. Source: https://zellij.dev/documentation/plugin-pipes.html
- Zellij plugin-to-plugin: `pipe_message_to_plugin(message_to_plugin: MessageToPlugin)` requires the `MessageAndLaunchOtherPlugins` permission. Source: https://zellij.dev/documentation/plugin-api-commands.html
- zjstatus regions and style: `format_left`, `format_center`, `format_right`, `format_space` select bar regions; `#[..]` carries directives (`fg`, `bg`, `bold`, `italic`, `underscore`, `blink`, `hidden`, `strikethrough`, `double-underscore`, `curly-underscore`, `dotted-underscore`, `dashed-underscore`, `reverse`); `color_` keys define aliases referenced as `$name`. Source: https://github.com/dj95/zjstatus/wiki/3-%E2%80%90-Configuration
- zjstatus overflow keys: `format_hide_on_overlength` and `format_precedence` govern crowded bars; frame keys are `hide_frame_for_single_pane`, `hide_frame_except_for_search`, `hide_frame_except_for_fullscreen`, `hide_frame_except_for_scroll`, `border_enabled`, `border_char`, `border_format`, `border_position`. Source: https://github.com/dj95/zjstatus/wiki/3-%E2%80%90-Configuration
- zjstatus notifications: publish with `zellij pipe "zjstatus::notify::<message>"` (no newlines); render with `{notifications}` configured by `notification_format_unread`, `notification_format_no_notifications`, `notification_show_interval`. Source: https://github.com/dj95/zjstatus/wiki/4-%E2%80%90-Widgets
- zjstatus command widgets: `{command_NAME}` with `command_NAME_command`, `command_NAME_format`, `command_NAME_interval`, `command_NAME_rendermode` (`static`, `dynamic`, `raw`), `command_NAME_cwd`, `command_NAME_env`. Source: https://github.com/dj95/zjstatus/wiki/4-%E2%80%90-Widgets
- zjstatus permissions: `ReadApplicationState` for mode/session/tab data, `ChangeApplicationState` for click actions, `RunCommands` for the command widget. Source: https://github.com/dj95/zjstatus/wiki/2-%E2%80%90-Permissions
- terminal-notifier: requires `-message`, `-remove`, or `-list`; `-group ID` replaces the previous notification in that group; `-remove ID|ALL`; `-list ID|ALL` prints tab-separated output; click routing is `-activate ID`, `-open URL`, or `-execute COMMAND`; `-sender ID` fakes the sending app and cannot combine with `-execute`/`-activate`. Source: https://github.com/julienXX/terminal-notifier
- alerter full roster (verified against the current README): `--message` (required), `--reply TEXT`, `--actions V1,V2,"V 3"`, `--dropdownLabel`, `--closeLabel`, `--title`, `--subtitle`, `--delay N`, `--at TIME`, `--timeout N`, `--sound NAME`, `--json`, `--group ID`, `--remove ID`, `--list ID`, `--sender ID`, `--appIcon PATH`, `--contentImage PATH`, `--ignoreDnd`. Source: https://github.com/vjeantet/alerter
- ntfy: click via the `Click` header or `ntfy publish --click=...`; actions via `Actions` or `--actions=...`; sequence mutation via publishing to `/<topic>/<sequence_id>`, the `X-Sequence-ID` header, or `ntfy pub --sequence-id=...`. Source: https://docs.ntfy.sh/publish/
- DuckDB JSON readers: `read_json(filename)` over a filename, list, or glob; `read_json_auto`, `read_ndjson`, `read_ndjson_auto` are aliases (`read_ndjson*` fix `format` to `newline_delimited`); parameters include `auto_detect`, `columns`, `dateformat`, `maximum_depth`, `records`, `sample_size`, `timestampformat`, `union_by_name`, `filename`, `format`, `ignore_errors`, `compression`, `maximum_object_size`, `hive_partitioning`, `maximum_sample_files`; `format` values are `auto`, `unstructured`, `newline_delimited`, `array`; `ignore_errors` works only with `newline_delimited`; since 1.3.0 `filename` also lands as an automatic virtual column. Source: https://duckdb.org/docs/current/data/json/loading_json
- DuckDB receipt-corpus form: `read_json([...paths or glob], filename = true, union_by_name = true)` composes multi-source schema drift with provenance; `COPY FROM ... (FORMAT json)` expects NDJSON by default with `ARRAY true` and `AUTO_DETECT true` variants. Source: https://duckdb.org/docs/current/data/json/loading_json
- DuckDB JSON processing: `json_transform(json, structure)` / `from_json(json, structure)` project nested `LIST`/`STRUCT` values with `_strict` variants; `json_each(json[, path])` and `json_tree(json[, path])` are lateral joins exposing `key`, `value`, `type`, `atom`, `id`, `parent`, `fullkey`, `path`, `json`, `root`, `rowid`. Source: https://duckdb.org/docs/current/data/json/json_functions
- jq tolerant JSONL admission: `tail ... | jq -n '[inputs | fromjson? | select(type == "object")]'` — the estate's proven live-feed rail. Sources: `modules/home/programs/shell-tools/mcp-launchers.nix:945-956` ; https://jqlang.org/manual/
- noti: `noti --pwatch 1234` watches a PID; `noti -t "Title" -m -` reads the message body from stdin. Source: https://codeberg.org/roble/noti/raw/branch/main/README.md

## [03]-[PATTERNS_WORTH_COMPOSING]

- One event stream, many renderers: the collector already converges hook-feed rows, process lanes, zjstatus pipe cells, and desktop posts; `{notifications}` joins as an additional renderer on the same stream, never a second event owner. Sources: `modules/home/programs/shell-tools/mcp-launchers.nix:938-1100` ; https://github.com/dj95/zjstatus/wiki/4-%E2%80%90-Widgets
- Plugin pipes as a message bus: sources are CLI, keybinding, or plugin; destinations are broadcast or URL-plus-configuration targeted with launch-on-miss; CLI pipes add backpressure and stdout replies — a full pub/sub substrate inside the multiplexer. Source: https://zellij.dev/documentation/plugin-pipes.html
- Status bars as widget format interpreters: region strings select widgets, per-widget keys parameterize rendering, `#[..]` carries style, aliases centralize color — configuration is a small language, not a template. Source: https://github.com/dj95/zjstatus/wiki/3-%E2%80%90-Configuration
- Notification class split by interaction contract: `terminal-notifier` owns replaceable async banners with click routing; `alerter` owns blocking question-and-answer alerts returning typed JSON; `UNUserNotificationCenter` is the first-party ceiling both wrap. Sources: https://github.com/julienXX/terminal-notifier ; https://github.com/vjeantet/alerter
- Log corpora as SQL tables: globs and file lists admit the whole receipt registry in one `read_json` call, `union_by_name` absorbs per-kind schema drift, and the `filename` virtual column preserves source identity — the registry JSON enumerates exactly the paths the query plane should glob. Sources: https://duckdb.org/docs/current/data/json/loading_json ; `modules/home/programs/shell-tools/browsers.nix:595`
- Registry-derived discovery: receipt sources are registry rows plus ssh-host-derived tunnel rows, so a new emitter becomes queryable by adding a row, never by patching a browser branch. Source: `modules/home/programs/shell-tools/browsers.nix:66-133`
- In-terminal operational views: receipt-query output renders inside the multiplexer as floating `viddy` panels rather than a separate GUI surface. Source: `modules/home/programs/apps/zellij/ops.nix:22-40`
- Cross-device semantics live in ntfy, not the local stack: remote subscribers, HTTP actions, priority routing, and mutable notification sequences are capabilities no macOS-local candidate expresses. Source: https://docs.ntfy.sh/publish/
- Process-boundary wrappers stay at process boundaries: noti's command wrap and PID watch notify on exit, but session attention requires joining hook events with live tty/process state — the collector's fold, which no wrapper replaces. Sources: https://codeberg.org/roble/noti/raw/branch/main/README.md ; `modules/home/programs/shell-tools/mcp-launchers.nix:938-957`

## [GAPS]

- `zjframes.wasm` ships beside `zjstatus.wasm` in every release; its capability surface is uninventoried — the next wave reads its docs and rules admission.
- The zjstatus `{pipe}` widget's full key roster beyond `pipe_NAME_format` and `pipe_NAME_rendermode` (interval, cwd, env analogues to the command widget) is unverified from the wiki.
- alerter packaging is unresolved: the README installs via a Homebrew tap; nixpkgs presence, code-signing/notarization posture, and TCC notification-permission behavior when the binary runs from a Nix profile are all unestablished — this decides whether alerter is admissible at all.
- terminal-notifier viability on current macOS is unproven: a 2017 upstream binary, unknown nixpkgs patch state, and unobserved behavior on macOS 26 — the estate depends on it today, so the next wave verifies the nixpkgs derivation and live posting path.
- ntfy estate fit is unresearched: the nixpkgs package identity, self-hosted server versus ntfy.sh custody, a Doppler-backed auth row, and the macOS receive-side client story.
- Whether the Forge DuckDB overlay build bundles or autoloads the `json` extension is unverified; an offline machine must not need an extension download at first query.
- The `UNUserNotificationCenter` delivery constraint is unestablished: whether actions and text-input require a signed app bundle, and by what mechanism alerter achieves sticky alerts on macOS 13+ — this bounds what any future first-party notifier can do from a CLI context.
- noti's nixpkgs state after the Codeberg move (which fork the derivation tracks) is unchecked — low priority given the non-adoption verdict.
- Whether a destination-less `zjstatus::notify::` broadcast reaches every running zjstatus instance (both estate bars) or requires targeted `--plugin` delivery pairs with the render-proof gap in `capabilities.md`.
